Tool Execution Workflow

Each MCP tool call is process-isolated. The wrapper script launches a fresh Node runtime for every invocation.

End-to-End Flow Diagram

Explorer UI / MCP client
    |
    | callTool("dpu_*", args)
    v
Ploinky Router -> AgentServer (/mcp)
    |
    | execute command from mcp-config.json
    v
dpuAgent/tools/dpu_tool.sh
    |
    | launch dpu_tool.mjs
    v
dpuAgent/tools/dpu_tool.mjs
    |
    | parse stdin envelope
    | normalize input and auth metadata
    | validate required fields and enums
    | dispatch to dpu-store operation
    v
dpuAgent/lib/dpu-store.mjs
    |
    | resolve actor identity and ACL
    | execute locked storage update/read
    v
JSON response -> MCP output -> Explorer UI

Detailed Lifecycle Stages

  1. Dispatch: AgentServer maps MCP tool name to wrapper command with TOOL_NAME.
  2. Envelope normalization: dispatcher extracts arguments from MCP input variants and auth from metadata.authInfo.
  3. Input validation: required strings and enum-like fields are validated before domain execution.
  4. Operation routing: tool name selects one domain function in dpu-store.mjs.
  5. Permission resolution: actor identity is normalized against manifest and ACL checks run before value access or mutation.
  6. Storage interaction: reads/writes occur through locked storage helpers and encrypted payload routines.
  7. Response emission: operation result is serialized as JSON; exceptions return ok: false payloads.

Runtime Modes

The same domain dispatch model is used for Ploinky runtime and standalone MCP server mode (server/standalone-mcp-server.mjs). This keeps contract behavior consistent while changing only hosting mode.

Mutating flows rely on lock-controlled storage writes to preserve atomic state updates.
